# 小红书图文草稿

## 标题备选

1. 我终于让 Codex 不再一个人硬扛复杂任务了
2. 把 Codex 用成一个小团队：我的子 agent 分工机制
3. AI 总是忘记用多线程？我给 Codex 写了一套防忘规则

## 开头

最近我在用 Codex 做比较复杂的项目时，发现一个问题：

它不是能力不够，而是经常用“单人模式”硬扛复杂任务。

比如一个任务明明可以拆成资料收集、代码实现、测试检查、最终整合，但它会自己一路做到底。即使我在线程开头说“请用多线程/子 agent”，后面聊着聊着它也可能忘掉。

所以我整理了一套机制，让 Codex 像一个小团队一样工作。

## 核心方法

第一步：先判断任务复杂度。

- 简单任务：不要开子 agent，直接做。
- 中等任务：可以派 1 个辅助 agent。
- 复杂任务：主线程做 Lead，再分派多个子 agent。

第二步：主线程必须当 Lead。

Lead 不是什么都亲自做，而是负责：

- 拆任务
- 派工
- 回收结果
- 处理冲突
- 最终整合

第三步：每个子 agent 都要有任务合同。

我会要求它写清楚：

- 这个 agent 负责什么
- 输入材料是什么
- 要交付什么
- 什么算完成
- 哪些文件不能碰

第四步：每轮都要回收。

我最常用的是三段：

- 派工摘要：这轮谁做什么
- 回收摘要：已经拿到什么
- 未回收项与缺口：还有什么没解决

这样 Codex 不容易“看起来做了很多，其实没法对账”。

## 我觉得最有用的一条

如果我在线程开头说：

```text
本线程启用多 agent 默认机制。
```

那么后续同一个线程里，它就要把“是否需要子 agent”当成持续偏好，不需要我每一轮都重复提醒。

这解决了我之前最头疼的问题：一开始说好了多线程，聊几轮以后它又忘了。

## 什么时候不要用

不要什么都上多 agent。

这些任务主线程直接做就好：

- 一句话解释
- 简单命令
- 单文件小改动
- 几分钟能完成的小任务

子 agent 是分工工具，不是魔法。

## 可复制模板

```text
本线程启用多 agent 默认机制。

每个非简单任务开始时，请先判断复杂度：
简单任务直接做；
中等任务必要时派 1 个辅助 agent；
复杂任务使用 Lead + 子 agent 并行。

如果分派子 agent，请先给派工摘要。
回收时请给回收摘要和未回收项。
主线程始终负责最终判断和整合。
```

## 结尾

我现在越来越觉得，AI 工作流的重点不是“让 AI 更努力”，而是给它一个稳定的协作结构。

复杂任务真正需要的不是一大段神奇提示词，而是一套能持续运转的分工、回收和纠偏机制。
