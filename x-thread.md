# X Thread Draft

1/  
我最近把 Codex 的使用方式从“单人助手”改成了“小团队协作”。

关键不是多开几个 agent，而是让主线程像 Lead 一样工作：判断复杂度、派工、回收、裁决、整合。

2/  
我遇到的核心问题：

复杂任务里，AI 很容易单线程硬扛。即使我一开始说“用多线程”，后续几轮它也可能忘掉。

所以我把“是否分派子 agent”写成了默认机制。

3/  
我的规则很简单：

简单任务：主线程直接做。  
中等任务：必要时派 1 个辅助 agent。  
复杂任务：Lead + 多个 Worker 并行。

不要为了多 agent 而多 agent。

4/  
主线程必须保持 Lead 职责：

- 拆任务
- 派工
- 回收结果
- 处理冲突
- 最终整合

子 agent 只做边界清楚的中间任务，不直接拍板。

5/  
每个子 agent 都要有任务合同：

- 角色
- 输入范围
- 交付物
- 完成标准
- 文件/模块所有权
- 禁止事项

这一步能显著减少“看起来很忙，但没法整合”的问题。

6/  
我最常用的回显结构：

- 派工摘要：谁做什么
- 回收摘要：拿到了什么
- 未回收项与缺口：还差什么

这比让 AI 一口气输出一大坨结论稳定很多。

7/  
另一个关键规则：

如果我在线程开头说过“本线程启用多 agent”，后续同一线程就把它当作持续偏好，不需要我每轮重复提醒。

这解决了“它过几轮又忘了”的问题。

8/  
我还加了防假多 agent 检查：

- 有没有真实 agent id？
- 有没有并行执行痕迹？
- 有没有明确交付物？
- 是否经过 Lead 整合？

没有这些，就只是换个说法的单线程。

9/  
这套机制最适合：

- 多文件代码改动
- 复杂资料研究
- PR/CI 修复
- 知识库整理
- 长文档审查

不适合一句话解释、简单命令、单文件小修。

10/  
我的结论：

Codex 的多 agent 能力真正有用的地方，不是“更多 AI 同时说话”，而是让任务变得可分工、可回收、可复盘。

AI 小团队的核心，仍然是一个清醒的 Lead。
